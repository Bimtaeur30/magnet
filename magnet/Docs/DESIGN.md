## 2. 게임 개요

| 항목 | 내용 |
|------|------|
| 가제 | **MAGNET SQUARE** |
| 장르 | 하이퍼 / 하이브리드 캐주얼 블록 퍼즐 |
| 한 줄 소개 | 자석 축에 블록이 자동 흡착되고, 정사각형을 완성하면 폭발하며, 매 턴마다 보드가 90° 회전하는 블록 퍼즐 |
| 레퍼런스 | Block Blast! (블록 퍼즐 골격), 무한의 계단 (점수 기반 스킨 해금) |
| 플랫폼 | 모바일(iOS/Android) 우선, **세로 화면·단일 핸드** |
| 시점 | **사이드뷰 2D** (격자 보드 정면) |
| 세션 길이 | 1~3분 |
| 팀 | 개발자 4명 (기획·아트 별도 인력 없음) |
| 프로토타입 목표 | **거의 전체 기능** — 코어 루프 + 점수 + 스킨 해금/인벤토리/장착/저장까지 |

### 2.1 핵심 차별점

1. **자석 자동 흡착** — x축 위치만 정하면 축 방향으로 이동해 첫 충돌 지점에 스냅. 정밀 드래그 없이 전략성 유지.
2. **매 턴 90° 회전** — 자석 축 중심으로 보드 전체(부착 블록 포함) 회전. 공간 추론이 매 턴 달라짐.
3. **정사각형 테두리 클리어** — 가로/세로 한 줄이 아니라, 자석 축을 중심으로 한 N×N(홀수) 테두리·내부 완성 시 폭발.

### 2.2 심사·방향성 메모

- 블록 퍼즐 장르 CPI 특성을 고려해, **접근성 높은 조작·즉각 재도전**을 우선한다.
- **스킨 해금**은 “다음 스킨까지 한 판 더” 리텐션 장치로 운용한다 (게임플레이 영향 없음).

## 3. 핵심 게임플레이 루프

한 턴(블록 1개 배치)의 **고정 순서** — 이 순서가 게임의 핵심 규칙이다.

```
블록 선택 → x축 배치 → 자석 흡착·부착
  → 경계 이탈 판정 (게임오버)
  → 정사각형 폭발 판정·처리·점수
  → 자석 축 90° 회전 (보드·블록 동반)
  → 다음 블록 후보 갱신
```

| 단계 | 설명 |
|------|------|
| 1. 블록 후보 | 화면 하단에 **3개** 무작위 블록 제시. 하나 선택 |
| 2. x축 배치 | 터치·드래그로 **좌우(x)만** 조정. 손을 떼면 흡착 시작 |
| 3. 자석 흡착 | 자석 축 방향으로 이동 → 경로상 **가장 먼저** 닿는 블록 또는 경계에 스냅 부착 |
| 4. 경계 이탈 | 부착 직후, 고정 경계 밖으로 나간 블록이 있으면 **즉시 게임오버** |
| 5. 폭발 판정 | 회전 **전** 좌표 기준. 완성된 정사각형 테두리(또는 내부 포함 전체) 폭발·제거·점수 |
| 6. 90° 회전 | 폭발 처리 후, 자석 축 기준 **시계방향 90°** (방향은 SO로 설정 가능). 모든 부착 블록 동반 회전 |
| 7. 반복 | 후보 1개 소모 시 새 블록으로 채움. 3개 모두 놓을 수 없으면 게임오버 |

> **주의:** `판정 → 폭발 → 회전` 순서는 절대 바뀌지 않는다.

## 4. 핵심 시스템

### 4.1 보드와 자석 축

- **N×N 정사각 격자** (프로토타입 기본값: `9`, ScriptableObject로 변경 가능).
- **자석 축**: 보드 정중앙 1칸에 고정. 이동하지 않음. 폭발·회전의 기준점.
- 자석 축 자체는 블록이 아님.

### 4.2 블록 공급

- 항상 **3개 후보** 동시 제시.
- 형태 풀 예시: `1×1`, `1×2`/`2×1`, `1×3`/`3×1`, `2×2`, `L자(3~4칸)` 등.
- 1개 사용 시 해당 슬롯만 새 블록으로 교체.
- **게임오버**: 3개 모두 어떤 위치에도 둘 수 없을 때.

| 형태 | 칸 수 | 난이도 | 비고 |
|------|-------|--------|------|
| 1×1 | 1 | 낮음 | 구원용 |
| 1×2 / 2×1 | 2 | 낮음 | 기본 라인 |
| 1×3 / 3×1 | 3 | 보통 | 테두리 변 |
| 2×2 | 4 | 보통~높음 | 내부 채움 |
| L자 | 3~4 | 높음 | 변수 창출 |

출현 가중치·난이도 곡선 수치는 플레이테스트 후 확정.

### 4.3 배치·자석 흡착

- 플레이어는 **x축 위치만** 결정. y(축 방향) 정착은 물리 규칙(첫 충돌)이 결정.
- 부착 후 플레이어가 직접 이동 불가. 위치 변경은 **회전**으로만 발생.
- **프리뷰(가이드 라인)** 권장 — 의도와 다른 정착 방지.

### 4.4 회전

- 블록 **1개 부착마다** 1회, 폭발 처리 **이후** 90° 회전.
- 회전으로 인한 경계 이탈은 별도 게임오버로 처리하지 않음 (부착 시점에 이미 걸러짐).

### 4.5 폭발(클리어) 판정

- 자석 축을 중심으로 한 **홀수 N×N** 정사각형 (3×3, 5×5, 7×7 …).
- **테두리** 칸이 끊김 없이 블록으로 채워지면 해당 테두리 블록 폭발.
- **내부까지** 모두 채워지면 테두리+내부 전체 폭발.
- 여러 크기가 동시에 만족하면 **동시 폭발**·점수 합산.
- 판정 시점: 부착 직후, **회전 전** 좌표.

### 4.6 게임오버 조건

1. 부착 직후 블록이 보드 경계 밖.
2. 3개 후보 모두 배치 불가.

### 4.7 점수

- 폭발 제거 칸 수에 비례한 기본 점수.
- 동시 다중 사각형 폭발 시 **멀티 클리어** 보너스.
- 연속 턴 폭발 시 **콤보 체인** 배율 (수치 TBD).
- 게임 종료 시 최종 점수 vs **베스트 스코어** 비교 노출.

### 4.8 스킨 시스템 (코스메틱)

| 기능 | 설명 |
|------|------|
| 해금 | 누적 점수(또는 베스트 스코어)가 **N점** 도달마다 새 스킨 해금 |
| 인벤토리 | 보유·잠금 상태 목록 UI |
| 장착 | 선택 스킨을 인게임 블록 비주얼에 적용 |
| 저장 | 보유 목록 + 현재 장착 스킨 **영구 저장** |
| 영향 | 게임플레이 수치에 **영향 없음** |

### 4.9 추후 확장 (프로토타입 범위 밖)

- 가변 회전 각도/방향 모드, 컬러 보너스, 자석 축 파워업, 일일 시드 경쟁, 광고 부스터·코인 등.

## 6. 기술 아키텍처 (프로젝트 규칙)

### 6.1 스택

| 영역 | 선택 |
|------|------|
| Unity | 6000.3.x, URP 2D |
| DI | **Reflect** |
| Async | **UniTask** (신규 Coroutine 금지) |
| Tween | **LitMotion** (DOTween 금지) |
| 이벤트 | **EventChannelSO** + `GameEvent` 파생 클래스 |
| 데이터 | **ScriptableObject** (보드 크기, 블록 형태, 스킨, 점수 테이블) |

### 6.2 책임 분리 (SOLID)

| 레이어 | 책임 |
|--------|------|
| Data | `BoardConfigSO`, `BlockShapeSO`, `SkinSO`, `ScoreConfigSO` |
| Domain / Rules | 격자 상태, 흡착 시뮬, 폭발 판정, 회전, 게임오버 — **순수 로직 우선** |
| Application | 턴 진행 오케스트레이션, 점수·스킨·세이브 |
| Presentation | 블록·보드 뷰, 이펙트, UI |
| Input | x축 드래그 → 배치 요청 |

- MonoBehaviour는 표현·입력·DI 바인딩. 규칙 판정은 테스트 가능한 클래스로 분리.
- 오브젝트 간 통신은 `EventChannelSO` (직접 C# event / UnityEvent 지양).

### 6.2.1 Reflex vs SerializeField (확정)

| 대상 | 연결 |
|------|------|
| ScriptableObject·에셋 참조 | Inspector `[SerializeField]` — **Reflex 사용 안 함** (프리팹 분리 시에도 유지) |
| 씬 내 다른 오브젝트·런타임 서비스 | Reflex `[Inject]` |

### 6.2.2 보드 좌표 (확정)

- **격자 좌표:** 자석 = **(0, 0)**. N=9 유효 보드 = `gx, gy ∈ [-4 .. 4]`. 자석 옆 = `(1, 0)`.
- **월드:** `GridToWorld` — `(gx * cellSize, gy * cellSize)`, 자석 = `Vector2.zero`.
- **변환:** `BoardCoordinates.GridToWorld` / `WorldToGrid` (+ `IsInBounds`).
- **점유 상태:** `BoardGrid` — `Dictionary<Vector2Int, bool>` (배열 없음). 보드 밖 칸도 키로 넣은 뒤 `IsInBounds`로 경계 이탈 판정 (Phase 4).

### 6.3 폴더·소유권

- 개인 코드: `Assets/MemberWorkspace/[username]/`
- **타 멤버 Workspace 수정 금지**
- 개인 작업 기록(Sequence): `Assets/MemberWorkspace/[username]/Docs/`
  - 인덱스: `SEQUENCE.md` · Phase 본문: `Sequence/phaseN.md` (Unity Project 창에서 열기)
- 공용 문서: `Docs/DESIGN.md`, `Docs/README.md`, `Docs/TODO.md`, `Docs/AI_COLLAB_GUIDE.md`
- AI 규칙: `CLAUDE.md` (루트), Cursor `.cursor/rules/main.mdc`
- 공용 런타임 코드 위치(예: `Assets/Shared/`)는 팀 합의 후 Phase 0에서 확정. 미정 시 각자 Workspace에 구현 후 통합.

### 6.4 주요 이벤트 (초안)

| 이벤트 | 발생 시점 |
|--------|-----------|
| `BlockPlacedEvent` | 블록 부착 완료 |
| `BoundaryViolationEvent` | 경계 이탈 게임오버 |
| `SquareClearedEvent` | 폭발 처리 (크기, 점수) |
| `BoardRotatedEvent` | 90° 회전 완료 |
| `ScoreChangedEvent` | 점수 갱신 |
| `SkinUnlockedEvent` | 스킨 해금 |
| `GameOverEvent` | 게임 종료 |

## 8. 구현 Phase (한 번에 하나만 진행)

에이전트·개발 모두 **Phase 단위**로 완료·검증 후 다음 Phase로 넘어간다.

### Phase 진행 규칙

1. 구현 전 **grill-me**로 구조·소유 클래스 확인 → **한국어 계획 제시** → **담당자 승인 후** 구현 (`CLAUDE.md` Phase 워크플로).
2. 한 Phase 완료 → **컴파일 에러 없음**(콘솔 확인) → 담당자 확인 → `Assets/MemberWorkspace/[이름]/Docs/Sequence/phaseN.md` 갱신 → 위 표 `✅` 갱신.
3. 다음 Phase는 담당자가 “시작”이라고 명시한 뒤, **1번(계획·승인)** 을 다시 거친다.
4. 에이전트는 Unity **플레이 모드에 직접 진입하지 않는다**. 런타임 확인은 담당자 플레이 후 콘솔 로그로 한다 (`CLAUDE.md` Debugging).
